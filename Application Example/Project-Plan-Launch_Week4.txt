Introduction to General Technical Concepts Distributed Systems
Principles Modern web applications, particularly those serving emergency
response functions, must be built on distributed systems principles to
ensure reliability under unpredictable load conditions. According to
Bass et al. (2013), software architecture must address quality
attributes including performance, availability, and scalability from the
earliest design phases. The Disaster Relief Resource Coordination System
(DRRCS) applies these principles through: Stateless Architecture: Each
backend instance maintains no session state, enabling horizontal scaling
through simple replication behind load balancers. This approach, derived
from Fielding’s (2000) REST architectural constraints, ensures that any
instance can handle any request without requiring session affinity.
Layered Separation: Following clean architecture principles (Martin,
2017), the system separates concerns into presentation, application,
domain, and infrastructure layers. This separation enables independent
testing, maintenance, and scaling of each layer. Concurrency Management:
Spring Boot’s thread pool architecture allows concurrent request
processing. However, improper thread configuration can lead to thread
starvation under load. The system implements tuned thread pools with
monitoring to prevent exhaustion. CAP Theorem Considerations In
distributed systems, the CAP theorem (Brewer, 2000) states that systems
must choose between Consistency, Availability, and Partition Tolerance.
DRRCS prioritizes availability and partition tolerance while maintaining
strong consistency for transactional operations such as resource
allocation. This trade-off acknowledges that during network partitions
(common in disaster scenarios), the system must remain operational even
if some data becomes temporarily inconsistent, with eventual consistency
restoring correctness post-partition. Scalability Modeling Scalability
in web applications follows predictable patterns. Richards and Ford
(2020) identify three scalability dimensions: Vertical Scaling: Adding
resources to existing servers (limited by hardware constraints)
Horizontal Scaling: Adding more server instances (theoretically
unlimited) Functional Decomposition: Splitting systems into services
that scale independently DRRCS implements horizontal scaling as its
primary scalability mechanism, with future microservices decomposition
planned as an extension path.

Project Development Process Agile Scrum Methodology DRRCS development
follows the Agile Scrum methodology, selected for its alignment with
research-oriented development and ability to accommodate evolving
requirements. Sommerville (2016) notes that Agile methods excel when
requirements are expected to evolve, as is typical in capstone projects
where learning informs implementation. Agile Scrum Development Process
The Scrum framework implements: Sprints: 2-week development cycles Daily
Stand-ups: 15-minute coordination meetings Sprint Planning: Task
selection and commitment Sprint Reviews: Demonstration of completed work
Retrospectives: Process improvement discussions Rationale for Scrum
Selection Alternative methodologies considered: Waterfall: Rejected due
to inflexibility when requirements evolve Kanban: Suitable but less
structured for time-boxed academic deliverables Extreme Programming:
Valuable practices adopted (pair programming, TDD) but full methodology
too intensive for two-person team Scrum provides sufficient structure
while accommodating the learning-oriented nature of capstone
development. Project Requirements and Specifications Requirements
engineering forms the foundation of successful software projects.
According to Wiegers and Beatty (2013), well-specified requirements
reduce project failure risk by 60%. DRRCS requirements are categorized
into: Functional Requirements: Specific behaviors and functions the
system must perform Non-Functional Requirements: Quality attributes
including performance, security, and usability Constraints: Technical,
budgetary, and temporal limitations Requirements were gathered through
stakeholder interviews, analysis of existing disaster response systems,
and review of emergency management literature.

Project Modeling and Design Architectural Design The system follows a
modular monolithic architecture with clear separation of concerns.
Newman (2021) argues that monolithic architectures remain appropriate
for systems where team size is small and domain boundaries are not yet
fully understood, with the option to decompose into microservices as the
system evolves.

Ethical Issues Related to the Project Ethical considerations in
emergency response systems extend beyond standard software ethics.
According to the ACM Code of Ethics (2018), computing professionals have
a responsibility to “contribute to society and human well-being.” In
disaster contexts, system failures can have life-or-death consequences.
Key ethical dimensions: Data Privacy: Emergency data may include
sensitive medical information Equitable Access: System must serve all
affected populations regardless of technical literacy Reliability as
Ethics: System failures during emergencies constitute ethical failures
Transparency: Users must understand how their data is used and decisions
are made

Project Title [5] Disaster Relief Resource Coordination System

Client/Organization Name [6] Illinois Emergency Management Agency
(IEMA) - Office of the Coordinator Note: This represents a simulated
client relationship for academic purposes. All stakeholder meetings and
requirements gathering were conducted with subject matter experts
familiar with emergency management operations. Client Contact: Sarah
Chen, Deputy Coordinator for Operations Organization Mission: Coordinate
disaster response across federal, state, and local agencies; allocate
resources; support affected communities during emergencies. Problem
Definition [7] Natural disasters create environments of extreme
operational pressure. Earthquakes, floods, hurricanes, and wildfires
often cause sudden surges in emergency assistance requests. During these
critical periods, response organizations must coordinate large volumes
of incoming requests for food, shelter, medical care, and volunteer
assistance within extremely short time frames. Current Challenges: The
Illinois Emergency Management Agency currently relies on a combination
of phone trees, spreadsheets, and email chains to coordinate disaster
response. This manual approach creates several critical problems:
Information Silos: Requests arrive through multiple channels (phone,
email, social media) with no centralized tracking, leading to duplicate
efforts and missed requests. Scalability Failures: During the 2023
tornado outbreak in northern Illinois, the existing system processed
only 40% of incoming requests within the critical first 24 hours,
delaying assistance to affected communities. Resource Misallocation:
Without real-time visibility into resource availability, volunteers and
supplies are often sent to locations where they are not urgently needed
while critical shortages persist elsewhere. No Audit Trail: Manual
coordination leaves minimal documentation, hampering post-disaster
analysis and improvement efforts. Security Vulnerabilities: Sensitive
personal and medical information shared through unsecured communication
channels creates privacy risks. The Gap: Existing commercial solutions
are either too expensive for government budgets (enterprise disaster
management software costing $50,000+) or too generic to address
emergency management workflows. Open-source alternatives lack the
reliability guarantees required for life-critical applications. Solution
Need: A scalable, secure, user-friendly platform that centralizes
emergency requests, coordinates resource allocation, and maintains
reliable operation during traffic surges—all within government budget
constraints.

Project’s Objectives [8] The Disaster Relief Resource Coordination
System aims to address the identified problems through the following
objectives: Primary Objectives: Centralized Request Management: Create a
unified platform where all emergency requests (food, shelter, medical,
volunteer) are captured, tracked, and visible to authorized responders
in real-time. Scalable Architecture: Design and implement a backend
system capable of handling 10x normal traffic during disaster spikes
without degradation, achieving <2 second response time at 1,000
concurrent users. Resource Optimization: Develop matching algorithms
that efficiently pair available resources (volunteers, supplies) with
emergency requests based on proximity, urgency, and resource type.
Role-Based Access Control: Implement granular security permissions
ensuring that sensitive data is only accessible to authorized personnel.
Real-Time Status Updates: Provide stakeholders with immediate visibility
into request processing status and resource allocation. Audit-Ready
Logging: Maintain comprehensive logs of all system activities for
post-disaster analysis and accountability. Secondary Objectives: Mobile
Responsiveness: Ensure the frontend interface functions effectively on
mobile devices used by field responders. Offline Capability: Enable
basic functionality during internet connectivity interruptions (future
enhancement). Integration Readiness: Design APIs that can integrate with
existing government systems (FEMA, state databases). Cost Efficiency:
Deliver the complete system within a $500 budget, demonstrating that
effective disaster response technology need not be expensive.

Importance of the Project [14] Why This Project Must Be Done
Humanitarian Imperative: During disasters, every minute counts. Research
shows that timely assistance reduces mortality rates by up to 40% in the
first 24 hours following a disaster (CRED, 2022). Current manual
coordination methods introduce dangerous delays that cost lives.
Economic Impact: The National Institute of Standards and Technology
estimates that inefficient disaster response costs the U.S. economy $15
billion annually in lost productivity and delayed recovery (NIST, 2021).
Improved coordination through technology can significantly reduce these
losses. Government Mandate: Executive Order 13653 (2013) requires
federal and state agencies to “improve the effectiveness and efficiency
of disaster response” through technology adoption. Many agencies,
including IEMA, lack the resources to implement these mandates without
accessible, affordable solutions. Technological Gap: While commercial
disaster management software exists, it remains inaccessible to smaller
agencies due to cost. The average enterprise solution costs
$50,000-$200,000 annually—far beyond typical state emergency management
budgets. This project demonstrates that effective solutions can be built
at a fraction of the cost. Academic Contribution: This project
contributes to the body of knowledge on scalable emergency response
systems, providing a reference implementation that other researchers and
practitioners can adapt and extend.

Ethical Issues [26] Data Privacy and Protection Emergency data includes
sensitive personal information: Names and contact details Medical
conditions and needs Exact locations of vulnerable individuals Family
compositions Ethical Safeguards: Data Minimization: Collect only data
necessary for emergency response Purpose Limitation: Data used only for
coordination, never for marketing or surveillance Access Control: Strict
role-based access with audit trails Encryption: AES-256 at rest, TLS 1.3
in transit Retention Policy: Data deleted after 90 days (unless required
for audit) Transparency: Clear privacy policy explaining data usage
Equity and Accessibility Emergency systems must serve all affected
populations: WCAG 2.1 AA compliance for users with disabilities
Mobile-first design for users with limited computer access Plain
language for users with varying literacy levels Multiple language
support (future enhancement) Reliability as Ethics During disasters,
system failures can cost lives: 99.9% availability target during
emergencies Graceful degradation under extreme load Manual override
procedures documented Regular load testing to verify capacity Informed
Consent Users must understand how their data is used: Clear consent at
registration Opt-out options for non-critical data Explanation of data
sharing with responders Bias Prevention Matching algorithms must not
discriminate: Regular audit of allocation patterns Transparency in
priority scoring Human override for algorithmic decisions
